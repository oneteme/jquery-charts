Référence des options

Toutes les propriétés de TableProvider<T> avec type et valeur par défaut.

TableProvider<T> est l'objet de configuration que vous passez au composant. Concrètement, c'est un objet TypeScript dans lequel vous décrivez comment votre tableau doit se comporter : quelles colonnes afficher, si la recherche est active, comment paginer les résultats, quels filtres proposer, etc. Le T représente simplement le type de vos données (ex : Intervention), ce qui permet à l'éditeur de détecter les erreurs au moment où vous écrivez le code.

Pourquoi jquery-table ?

Les points concrets qui font la différence au quotidien.

  • 01
    Slice by et Group by intégrés

    Filtres par facettes et regroupement de lignes configurables en quelques lignes de config, sans logique dans le composant parent. Les panneaux de slice et les en-têtes de groupe sont gérés entièrement par le composant — zéro state management externe.

  • 02
    Syntaxe unifiée avec jquery-charts

    TableProvider<T> suit exactement le même contrat que ChartProvider<T> : un seul objet de config passé en [config], les données séparées en [data]. Un développeur qui connaît l'un maîtrise immédiatement l'autre — courbe d'apprentissage quasi nulle.

  • 03
    Très performant sur de grands volumes

    Le composant utilise ChangeDetectionStrategy.OnPush et ne recalcule la vue que lorsque les références de données changent. Tri, filtres, group by et pagination sont résolus en un seul passage sur les données — pas de recalcul en cascade à chaque interaction.

  • 04
    Templates sur mesure via jqtCellDef

    N'importe quelle cellule peut être remplacée par un <ng-template> Angular natif. Le composant reste maître du tri, de la pagination et de la recherche ; le développeur contrôle uniquement le rendu visuel, colonne par colonne.

  • 05
    Typage fort end-to-end

    Le générique T traverse tout : définition des colonnes, value(), sortValue(), rowClass(), onRowSelected(). L'éditeur détecte toute incohérence au moment où tu écris, pas à l'exécution.

  • 06
    Préférences utilisateur persistées sans effort

    Un seul champ preferences: { tableId: '...' } active la persistance complète : ordre des colonnes, largeurs, group by, filtres slice — tout est restauré au rechargement, sans écrire une ligne de localStorage.

  • 07
    Colonnes lazy out-of-the-box

    Le champ lazy.fetchFn suffit pour différer le chargement d'une colonne coûteuse. Le composant gère le spinner, l'état d'erreur et l'intégration dans le tri — sans logique supplémentaire dans le parent.
